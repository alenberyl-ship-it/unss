# 📝 CHANGELOG - UNSS Goma v2.1 Sécurité & Production Ready

---

## 🔴 CRITIQUES - Vulnérabilités Fixées

### 1. Path Traversal dans `/download`
**Severité**: 🔴 CRITIQUE  
**Avant**: Possibilité d'accéder aux fichiers en dehors du dossier uploads  
**Après**: 
- Normalisation du chemin
- Vérification contre répertoire de base
- Rejet des fichiers non autorisés
- Logging des tentatives

### 2. Credentials Admin par Défaut
**Severité**: 🔴 CRITIQUE  
**Avant**: Admin créé automatiquement toujours (admin@sauveteurs.com/admin123)  
**Après**:
- Admin par défaut SEULEMENT en développement
- Production: création manuelle obligatoire
- Avertissement sur login de démo

### 3. Configuration Sécurité Production
**Severité**: 🔴 CRITIQUE  
**Avant**: `debug=True` toujours, `FLASK_ENV` ignoré  
**Après**:
- `FLASK_ENV` contrôle le mode
- `debug=False` automatiquement en production
- Sessions Secure/HTTPOnly en production
- Validation des variables d'env obligatoires

### 4. Injection/Traversal Fichiers
**Severité**: 🟠 HAUTE  
**Avant**: Pas de validation type fichier, pas de limite size  
**Après**:
- Séparation ALLOWED_IMAGE_EXTENSIONS et ALLOWED_PDF_EXTENSIONS
- Vérification size par fichier avant save
- Sanitization noms de fichiers
- Hash dans les noms pour collision prevention

---

## 🟠 HAUTE - Améliorations Sécurité

### 5. Validation des Entrées
**Avant**: Validation minimale, pas de regex  
**Après**:
- Validation email stricte (regex pattern)
- Limite de longueur strings
- Validation password minimum (8 chars)
- Sanitization caractères dangereux

### 6. Logging & Audit
**Avant**: Pas de logging  
**Après**:
- Logging complet à `/logs/app.log`
- Tentatives connexion échouées loggées
- Accès non autorisés loggées
- Actions admin loggées (ajout/suppression)
- Téléchargements loggés

### 7. Gestion d'Erreurs
**Avant**: Stack traces exposés aux users  
**Après**:
- Messages d'erreur génériques pour users
- Logging détaillé côté serveur
- Pages d'erreur custom (404, 500, 403)
- Exception handling partout

### 8. HTTP Security Headers
**Avant**: Aucun header de sécurité  
**Après**:
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: SAMEORIGIN`
- `X-XSS-Protection: 1; mode=block`
- `Strict-Transport-Security` (production)

---

## 🟡 MOYENNE - Améliorations Structurelles

### 9. Configuration Production
**Avant**: Hardcodage, pas de config centralisée  
**Après**:
- Fichier `config.py` centralisé
- Support pour dev/prod/test environments
- Variables d'env validées
- Configuration Gunicorn

### 10. Sessions
**Avant**: Sessions basiques, pas sécurisées  
**Après**:
- `SESSION_COOKIE_SECURE` en production
- `SESSION_COOKIE_HTTPONLY` toujours
- `SESSION_COOKIE_SAMESITE` = Lax
- Timeout 24h configuré
- Session.clear() au logout

### 11. Email
**Avant**: Configuration dans fichier hardcoder  
**Après**:
- Tous les credentials en variables d'env
- Configuration obligatoire validation
- Headers Reply-To définis
- No credentials exposed

### 12. Database
**Avant**: DATABASE_NAME fixe  
**Après**:
- DATABASE_NAME configurable via .env
- Logging des opérations
- Fonction get_admin_par_contact added
- Compter admins function

## 🔄 BREAKING CHANGES

### Pour les Développeurs:
1. `FLASK_ENV` doit être défini pour mode production
2. `admin_required` decorator maintenant complet (plus besoin de vérification manuelle)
3. `allowed_file()` signature changée (prend extension list)

### Pour les DevOps:
1. Gunicorn recommandé obligatoire (pas Flask dev server)
2. Variables d'env obligatoires en production
3. `.env` doit exister (copy depuis `.env.example`)
4. Logs dossier doit exister

### Aucun breaking change pour les Users
L'app reste compatible, juste plus sécurisée.
